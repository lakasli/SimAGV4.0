# 目标
把系统强制切成L1/L2/L3/L4四层，并写出“不可违背”的依赖与调用规则，让后续所有功能都能按同一纪律扩展与Debug。

# 关键铁律（必须写进项目规范）
- 依赖方向：L1 -> L2 -> (L3/L4)
- 同层禁止互调：L4原子之间禁止互调；L3分子之间禁止互调
- 下层禁止调用上层；禁止循环依赖
- 控制流与业务逻辑分离：流程/路由只在L2；实现细节在L3/L4

# 交付物（你要输出这些）
1) 分层目录与命名约定（按项目语言/框架适配）
2) 模块类型定义：Atom、Molecule、Coordinator、Entry 的职责边界
3) 接口契约：L2如何调用L3/L4（函数/消息/命令对象）
4) 依赖检查清单：PR评审项 + 静态检查策略建议（不写具体工具也可）

# 工作流
1. 询问/推断：项目语言、运行形态（CLI/API/服务/嵌入式）、是否已有模块边界
2. 生成目录骨架：给出每层的职责与“允许包含的内容”
3. 定义接口契约：用“命令/端口（ports）”抽象L2对下层调用
4. 定义横切角色：外交官（通信）、数据员（数据路由）、指挥官（流程表）
5. 输出评审清单与违例示例（帮助团队对齐）

# 输出模板（可直接套用）
## 目录建议
- L1: entry/
- L2: coordinator/
- L3: molecules/
- L4: atoms/

## 模块契约（示例结构）
- atoms: 纯函数/小模块；不引用其他atom；可单测
- molecules: 组合多个atom；不引用其他molecule；可集成测
- coordinator: 只负责流程表、路由、通信、数据编排；不实现业务细节
- entry: 解析输入/协议适配；只调用coordinator

## PR检查清单
- 是否出现同层互调？
- 是否出现下层调用上层？
- 控制流是否被下沉到L3/L4？
- 模块是否可单独测试与理解？
