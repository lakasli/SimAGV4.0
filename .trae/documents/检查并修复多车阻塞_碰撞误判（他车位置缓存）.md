## 现状结论（基于你给的两个 state）
- AMB-09/AMB-11 的 state 里出现的是 54231（Caution: robot is blocked），这是引擎侧 `navigationBlocked_` 触发的“边未释放等待”，不是车车碰撞/雷达阻挡链路。
- 如果是车车阻挡/碰撞链路生效，state.errors 会出现 54330/54331，并且 visualization 里会带 `overlaps.radar/safety` 指向阻挡来源。

## 碰撞/阻挡链路梳理（需要核查的缓存点）
- 他车缓存入口：MQTT 收到 `.../visualization` → `SimInstanceCoordinator::handleVisualization()` 解析 `manufacturer/serialNumber` + `agvPosition.mapId` + `safety(center,length,width,theta)` → 写入 `otherVisualizationsByKey_`。
- 判定入口：每 tick `tickOnce()` 清理 15s 过期缓存 → 构造自车 radar 扇形与 safety 矩形 → 对缓存内每台车做：
  - 距离粗筛：`dist2 <= (radarRadius*10)^2`
  - 同图过滤：当前逻辑仅在“双方 mapId 都非空且不相等”时才跳过
  - SAT 相交：safety-safety → collision；radar-safety → blocked

## 需要改的关键点（降低“远距离误判/跨地图误判”）
1) 修正跨地图过滤：
- 现逻辑：只有双方 mapId 都非空才比较；一旦任意一方 mapId 为空，会把对方当“同图”参与判定，可能造成跨地图误判。
- 拟改：只要双方 mapId 有一方非空，则必须相等才参与判定（否则跳过）。

2) 增强缓存有效性校验：
- 丢弃 safety 尺寸非正、坐标/角度非有限值（NaN/Inf）的消息。
- 可选：当缓存条目的 mapId 为空时，直接标记为“不可参与碰撞判定”，仅保留用于调试。

3) 增强调试可观测性（用于你们快速对照“到底被谁挡住”）：
- 在 trace 的 `applyPerception` 附加 blockedBy/collidedWith 的具体 key 列表（或前 N 个），以及最近一次命中的距离/地图。
- 可选：把 54330/54331 的 peer 写入 `errors[].errorReference`（state 层直接看到是谁导致）。

## 验证方式（改完后如何确认问题消失）
- 构造两车不同 mapId/或一车 mapId 为空的 visualization：应不再触发 54330/54331。
- 构造两车同 mapId、相距很远（大于 gateRadius）：应不触发 blocked/collision。
- 真实联调：观察每台车 trace 中 `applyPerception blockedBy=[...]` 与 visualization.overlaps 是否一致；同时确认 54231（edge released 等待）与 54330（雷达被车挡）能区分。
